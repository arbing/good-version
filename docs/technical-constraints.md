# 好版本技术实现约束

## 技术选型

- 桌面框架：Tauri
- 前端：Web 技术栈
- 后端能力：Tauri Rust 侧实现文件系统与 Git 操作
- Git 能力：应用内置，不要求用户安装系统 Git
- 首版平台：macOS + Windows
- 首版语言：中文

## Git 使用约束

界面完全不暴露 Git 概念，底层可以使用 Git。

如果项目没有 `.git/`：

- 初始化本地 Git 仓库
- 不配置远端
- 创建初始好版本

如果项目已有 `.git/`：

- 复用已有 `.git/`
- 只操作当前分支
- 不切换分支
- 不创建业务分支
- 不合并
- 不推送
- 不修改远端配置
- 不修复外部 Git 操作造成的复杂状态

## 保存实现约束

每次保存「当前好版本」必须：

1. 收集可保存文件
2. 更新 Git 索引
3. 创建 commit
4. 创建内部 tag
5. 写入应用元数据

内部 tag 用于保证 hard reset 后版本仍可找回。

建议 tag 命名规则：

```text
refs/tags/good-version/<version-id>
```

应用元数据至少记录：

- versionId
- commitHash
- tagName
- createdAt
- title
- note
- fileChangeSummary
- fileList
- isRollbackCheckpoint

## 回退实现约束

回退必须先保存当前状态。

流程：

1. 创建「回退前状态」commit
2. 创建对应内部 tag
3. 确认保存成功
4. 确认目标 tag 存在
5. 执行 hard reset 到目标 commit
6. 更新时间线当前状态

禁止在当前状态保存失败时执行 hard reset。

## 文件保存约束

MVP 保存所有用户要求保存的内容：

- 保存大文件到普通 Git
- 保存 `.env`
- 不提示敏感文件风险
- 不自动启用 Git LFS

工具可以有内部排除规则，但不随意修改用户 `.gitignore`。

内部排除规则只应用于明确无意义或工具自身临时文件。

## 远端约束

工具绝不自动推送。

工具不得：

- `git push`
- 修改 remote
- 自动创建远端仓库
- 自动登录 GitHub
- 自动上传项目文件

如果已有远端，保存 commit 造成当前分支 ahead，MVP 接受该状态，界面不展示 Git 术语。

## 元数据约束

时间线不能只依赖当前 Git 线性历史。

原因：hard reset 后部分 commit 不再从 HEAD 可达，但仍要在时间线中显示。

必须通过内部 tag 和应用元数据保留所有好版本记录。

## 导出约束

导出当前项目副本时：

- 包含 `.env`
- 包含当前工作区全部项目文件
- 排除 `.git/`
- 排除应用元数据
- 排除工具内部临时文件
- 不包含版本历史

## 安全与风险边界

MVP 的安全边界是本地保护，不是密钥管理。

已确认的产品决策：

- 静默保存 `.env`
- 有远端时也保存 `.env`
- 不做风险提示
- 绝不自动推送

因此实现必须严守“不上传、不推送、不联网同步”。
