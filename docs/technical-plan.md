# 好版本技术方案

## 1. 架构概览

「好版本」采用 Tauri 桌面应用架构：

- 前端负责项目列表、时间线、确认弹窗、变化摘要展示
- Rust 侧负责文件系统、Git 操作、导出、存储统计
- Git 作为底层版本存储引擎
- 应用元数据用于维护非线性时间线和用户说明

## 2. 核心模块

### 项目管理模块

职责：

- 添加项目文件夹
- 读取项目列表
- 修改项目显示名
- 打开项目文件夹
- 判断项目是否已有 `.git/`
- 为无 `.git/` 项目创建外置 Git 数据目录
- 记录工作区路径和版本库路径
- 项目目录丢失时恢复工作区

### 版本保存模块

职责：

- 收集当前文件状态
- 应用内部排除规则
- 更新 Git 索引
- 创建 commit
- 创建内部 tag
- 生成变化摘要
- 写入版本元数据

### 时间线模块

职责：

- 读取所有版本元数据
- 校验内部 tag 是否存在
- 标记当前版本
- 按时间倒序展示版本卡片

### 回退模块

职责：

- 回退前创建「回退前状态」
- 校验保存成功
- 解析目标版本 commit
- hard reset 到目标版本
- 更新时间线状态

### 导出模块

职责：

- 导出当前项目副本
- 复制当前项目文件
- 排除 `.git/` 目录
- 排除 `.git` 指针文件
- 排除工具元数据
- 保留 `.env`
- 生成项目副本
- 导出某个好版本的版本压缩包
- 从目标好版本对应的版本对象读取文件并写入 zip
- 版本压缩包内包一层项目目录
- 不修改当前项目目录

### 存储统计模块

职责：

- 计算外置 Git 数据目录或项目内 `.git/` 占用大小
- 计算项目目录占用大小
- 展示版本占用估算

## 3. 数据模型

### Project

```ts
type Project = {
  id: string
  displayName: string
  path: string
  gitDirPath: string
  usesExternalGitDir: boolean
  createdAt: string
  updatedAt: string
}
```

### Version

```ts
type Version = {
  id: string
  projectId: string
  title: string
  note?: string
  commitHash: string
  tagName: string
  createdAt: string
  isInitial: boolean
  isRollbackCheckpoint: boolean
  changeSummary: ChangeSummary
}
```

### ChangeSummary

```ts
type ChangeSummary = {
  added: number
  modified: number
  deleted: number
  files: Array<{
    path: string
    status: 'added' | 'modified' | 'deleted'
  }>
}
```

## 4. Git 操作策略

### 初始化项目

无 `.git/` 时：

1. 在应用数据目录创建外置 Git 数据目录
2. 将项目目录作为工作区
3. 创建初始好版本 commit
4. 创建内部 tag
5. 写入项目和版本元数据

已有 `.git/` 时：

1. 读取当前分支和当前 HEAD
2. 不切换分支
3. 不修改远端
4. 创建初始好版本 commit
5. 创建内部 tag

### 保存好版本

1. 扫描工作区
2. 应用内部排除规则
3. stage 应保存文件
4. 创建 commit
5. 创建内部 tag
6. 计算变化摘要
7. 写入版本元数据

### 回退版本

1. 创建回退前状态版本
2. 确认 commit 和 tag 已创建
3. 从目标版本 tag 解析 commit
4. 执行 hard reset
5. 更新时间线当前状态

## 5. 元数据存储

应用需要存储项目列表和版本列表。

项目元数据必须记录工作区路径、Git 数据目录路径，以及是否使用外置 Git 数据目录。

好版本创建的新仓库，Git 数据目录保存在应用数据目录中，项目目录只作为工作区。已有 Git 项目继续复用项目内 `.git/`。

版本可找回性由 Git tag 保证。项目复制到另一台电脑后，如果只复制工作区，不会包含好版本保存的版本历史；后续可考虑提供包含版本历史的迁移导出。

## 6. 排除规则

MVP 不随意修改用户 `.gitignore`。

内部排除规则只排除：

- 工具自身临时文件
- 操作系统临时文件
- 明确不应进入版本的中间文件

`.env`、大文件、构建产物在当前产品决策下默认保存。

## 7. 错误处理

### 保存失败

- 显示保存失败原因
- 不创建时间线卡片
- 不执行后续回退

### 回退前保存失败

- 阻止 hard reset
- 提示用户当前状态未保存，不能回退

### 目标版本丢失

- 如果 tag 不存在，禁用「回到这里」
- 提示该版本无法找回

### 项目目录丢失

- 如果项目使用外置 Git 数据目录，提示用户选择恢复位置
- 目标位置为空目录或不存在时才执行恢复
- 恢复成功后更新项目路径元数据
- 如果项目复用已有 `.git/`，提示项目文件夹不可用

### Git 仓库异常

- 首版只提示项目状态异常
- 不自动修复外部 Git 操作

## 8. 实施顺序

### 阶段 1：基础项目与版本保存

- Tauri 项目骨架
- 项目列表
- 添加项目
- 创建外置 Git 数据目录或复用已有 `.git/`
- 保存当前好版本
- 创建 tag

### 阶段 2：时间线与变化摘要

- 版本元数据
- 时间线卡片
- 文件变化摘要
- 当前版本标记

### 阶段 3：回退闭环

- 二次确认
- 回退前状态保存
- hard reset
- 回退前状态可找回

### 阶段 4：导出与存储占用

- 导出当前项目副本
- 导出指定好版本的版本压缩包
- 排除 `.git/` 目录和 `.git` 指针文件
- 展示存储占用

### 阶段 5：体验打磨

- 中文文案
- 空状态
- 错误提示
- 低压力视觉风格
