目标：建立可运行的桌面应用基础。

任务：

- [x] 初始化 Tauri + 前端项目。

- [x] 建立前端页面结构。
- [x] 建立 Rust command 调用边界。
- [x] 建立应用数据目录读写能力。

## 3. Git / 版本服务模块

### 3.1 Git 存储模式识别

目标：根据项目是否已有 `.git` 选择不同实现方式。

任务：

- [x] 判断项目目录是否存在 `.git`。
- [x] 已有 `.git`：复用当前仓库。
- [x] 无 `.git`：创建外置 Git 数据目录。
- [x] 外置 Git 不污染项目目录。
- [x] 不切分支、不推送、不合并。

进度：已完成第一版存储模式识别和保存边界，待用真实目录验证外置数据目录行为。

验收：

- 普通非 Git 项目可以被版本化。
- 已有 Git 项目不会被切换分支。
- UI 不出现 Git、commit、branch、tag 等术语。

### 3.2 自动保存初始好版本

目标：添加项目后立即创建第一个可恢复版本。

任务：

- [x] 添加项目成功后自动执行保存。
- [x] 生成版本元数据。
- [x] 创建内部版本标识。
- [x] 标记为“初始好版本”。

进度：已完成初始好版本保存，回退能力在第二阶段实现后补验。

验收：

- 新项目添加后时间线至少有一个版本。
- 初始版本可被回退。
- 保存失败时项目添加流程能给出明确错误。

### 3.3 手动保存好版本

目标：用户能在项目可用时保存当前状态。

对应原型：

- 主界面保存按钮
- 初始好版本页面

任务：

- [x] 实现“保存当前好版本”按钮。
- [x] 支持填写可选说明。
- [x] 自动生成时间标题。
- [x] 计算文件变化摘要。
- [x] 保存底层版本记录和应用元数据。

进度：已完成手动保存第一版，无变化时返回中文提示且不创建新版本。

建议版本数据：

```ts
type Version = {
  id: string
  projectId: string
  title: string
  description?: string
  createdAt: string
  internalRef: string
  changeSummary: {
    added: number
    modified: number
    deleted: number
    files: Array<{
      path: string
      status: "added" | "modified" | "deleted"
    }>
  }
}
```

验收：

- 保存后时间线新增版本。
- 可选说明能正确展示。
- 文件新增、修改、删除数量正确。
- 没有变化时给出友好提示，不制造空版本。

### 3.4 版本时间线

目标：用户能按时间查看所有好版本。

对应原型：

- 综合主界面
- 回退到中间版本页面

任务：

- [x] 展示版本标题、时间、说明、变化数量。
- [x] 标记当前所在版本。
- [x] 支持选择某个版本查看详情。
- [x] 时间线来源以应用元数据为准，不依赖线性 Git 历史。

进度：已完成时间线展示、当前标记和版本变化详情入口。

验收：

- 多个版本按时间倒序展示。

- 回退后时间线仍完整。

## 5. 回退模块

### 5.1 回退确认弹窗

目标：降低误操作风险。

任务：

- [x] 点击“回到这里”后弹出确认。
- [x] 说明会先保存当前状态。
- [x] 明确目标版本时间和说明。
- [x] 支持取消。

进度：已完成回退确认弹窗，文案不暴露底层技术概念。

验收：

- 用户必须二次确认才能回退。
- 弹窗文案不使用 Git 术语。
- 取消后项目状态不变。

### 5.2 回退前自动保存

目标：保证回退操作可恢复。

任务：

- [x] 回退执行前自动保存当前状态。
- [x] 标题建议为“回退前状态”。
- [x] 记录完整版本元数据。
- [x] 当前无变化时，提示无需额外保存或复用当前版本。

进度：已完成回退前自动保存；无变化时直接执行回退，不制造空版本。

验收：

- 回退前状态出现在时间线中。
- 用户可以从旧版本再回到回退前状态。
- 自动保存失败时中止回退。

### 5.3 执行回退

目标：把项目文件恢复到目标好版本。

任务：

- [x] 校验目标版本内部标识存在。
- [x] 执行安全的工作区恢复。
- [x] 更新当前版本标记。
- [x] 回退完成后刷新时间线和变化状态。

进度：已完成指定好版本回退，底层恢复失败时返回错误并避免更新当前标记。

验收：

- 文件内容恢复到目标版本。
- 回退后能继续保存新版本。
- 目标版本丢失时给出错误，不破坏当前文件。

- [x] 保留 `.env`、构建产物、大文件。

### 6.2 导出版本压缩包

目标：用户能把任意一个好版本导出为不带版本历史的 zip 压缩包。

任务：

- [x] 在每个版本卡片增加“导出压缩包”入口。
- [x] 选择导出目标目录。
- [x] 根据项目名、版本说明和日期生成默认 zip 文件名。
- [x] 同名 zip 已存在时自动生成不重名文件。
- [x] 从目标好版本对应的版本对象读取文件并写入 zip。
- [x] zip 内包一层项目目录。
- [x] 保留 `.env`、隐藏文件和普通项目文件，排除 `.git` 版本控制数据。
- [x] 导出成功后打开导出目录。

进度：已完成第一版实现，决策记录见 `docs/adr/0001-export-version-archive-from-version-object.md`。

验收：

- 导出的 zip 解压后包含对应好版本的项目文件快照。
- 导出不会修改当前项目目录。
- 外置保存数据项目在项目目录丢失时仍可导出历史好版本。
- 复用项目内版本数据的项目在目录丢失时给出友好提示。
- 重复导出同一版本会生成多个不重名 zip。

## 12. MVP 验收主路径

最终 MVP 至少要通过这条路径：

1. 打开应用。
2. 添加一个本地项目。
3. 自动生成初始好版本。
4. 修改项目里的文件。
5. 保存当前好版本。
6. 再修改或删除文件。
7. 查看版本变化摘要。
8. 选择旧版本并确认回退。
9. 系统先保存回退前状态。
10. 项目文件恢复到旧版本。
11. 再选择“回退前状态”恢复回来。
12. 导出当前项目副本。
13. 导出的副本不包含版本历史，但包含 `.env`。
14. 导出某个好版本的版本压缩包。
15. 解压后的版本压缩包包含对应好版本的项目文件快照。
